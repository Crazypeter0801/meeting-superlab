## Clarification Questions

```text
我先不直接出方案，先把需求问清楚。请尽量回答下面几个关键问题：

1. 主要用户是谁：主持人、参会者、外部嘉宾、企业管理员，还是其他角色？
2. 问题发生在哪个阶段：会前、会中、会后、管理端、跨设备，还是异步协作？
3. 当前用户路径或产品行为是什么？
4. 现在最核心的痛点是什么：效率、理解、信任、控制、协作、转化，还是其他？
5. 希望这次探索改善什么结果？
6. 有哪些约束需要先考虑：权限、隐私、AI 准确性、跨端一致性、企业规则、排期、现有组件？
7. 你手上有什么材料：截图、PRD、会议纪要、数据、用户反馈、竞品链接？
8. 这一轮你希望产出什么：需求澄清、竞品分析、方案方向、demo prompt，还是评审材料？
```

## Competitor Research Request

```text
请围绕这个会议/协作体验需求做竞品启发分析。

优先搜索公开资料，包括官网、帮助文档、发布说明、支持文档、可信评测。
如果公开资料不足，请明确告诉我需要体验哪些竞品、截哪些页面、记录哪些信息。

参考竞品可包括：
Zoom、Microsoft Teams、Google Meet、Webex、飞书/Lark、钉钉、Slack、Telegram、Discord，以及 AI 会议工具如 Fireflies、Otter.ai、Read AI、Granola。

请区分事实观察、推断和建议。
```

## Low-Fidelity Demo Prompt

```text
请基于以下方案生成会议产品体验的黑白低保真 demo。

目标：

用户角色：

使用场景：

核心流程：

页面/状态：
1. 正常状态
2. 空状态
3. 权限或异常状态
4. 多角色或跨端状态

设计要求：
- 黑白低保真
- 不要过度视觉风格
- 聚焦信息结构、操作路径、关键状态
- 使用真实但简洁的会议产品文案
- 可以借鉴竞品的交互思路，但不要复制竞品视觉

请输出页面结构、关键控件、用户路径和评审时需要关注的问题。
```

## Figma Handoff Question

```text

如果需要，请发我：
1. Figma 文件或具体 node 链接
2. 希望新增页面、放在某个节点旁边，还是更新已有稿
3. 画稿范围：Tiny flow / Review-ready flow / Concept flow
4. 保持黑白低保真，还是做更接近腾讯会议产品稿的样式
5. 是否需要复用文件里的现有组件或设计系统
```
